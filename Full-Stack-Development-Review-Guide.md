# 🚀 Full-Stack 커뮤니티 프로젝트: 신입 개발자를 위한 온보딩 & 상세 복습 가이드

본 가이드는 지난 5일간 구축한 **'Kafka 기반 실시간 커뮤니티 플랫폼'**의 핵심 개념과 동작 원리를 입문자의 시선에서 상세히 설명합니다. 이 프로젝트는 단순한 게시판을 넘어, 현대적인 대규모 분산 시스템의 기초를 모두 담고 있습니다.

---

## 🏗 1. 전체 아키텍처 개요: "어떻게 데이터가 흐르는가?"

풀스택 개발의 핵심은 **'데이터의 흐름'**을 이해하는 것입니다. 우리 프로젝트는 크게 세 가지 통신 방식을 사용합니다.

1.  **Request-Response (HTTP REST API):** 클라이언트가 요청하고 서버가 응답하는 방식 (예: 글 목록 가져오기).
2.  **Server-Sent Events (SSE):** 서버가 클라이언트에게 일방적으로 알림을 쏘는 방식 (예: 새 글 알림).
3.  **Bidirectional (WebSocket/STOMP):** 서버와 클라이언트가 서로 자유롭게 대화하는 방식 (예: 채팅).

이 모든 것을 하나로 묶어주는 **심장** 역할이 바로 **Apache Kafka**입니다.

---

## 📚 2. 일차별 핵심 기능 상세 분석

### [Day 1] 게시판의 기초: JPA와 CRUD
가장 먼저 '데이터를 저장하고 보여주는' 기본 기능을 만들었습니다.

* **핵심 기술:** Spring Data JPA, MySQL
* **동작 원리:**
    * `Post` 엔티티는 데이터베이스의 테이블과 1:1로 매핑됩니다.
    * `Repository` 인터페이스를 통해 복잡한 SQL 쿼리 없이도 데이터를 저장(`save`)하고 조회(`findAll`)할 수 있습니다.
* **복습 포인트:** "왜 DTO를 쓰는가?" -> 엔티티를 직접 화면에 노출하면 보안에 취약하고 변경에 유연하지 못하기 때문입니다.

### [Day 2] 실시간 알림: Kafka와 SSE의 만남
누군가 글을 썼을 때, 접속 중인 모든 유저에게 즉시 알림을 띄우는 기능입니다.

* **핵심 기술:** Apache Kafka, Server-Sent Events (SSE)
* **동작 원리:**
    1.  **글 저장:** DB에 글이 저장됩니다.
    2.  **이벤트 발행 (Producer):** 서버가 Kafka의 `community.new-post` 채널에 "새 글이 올라왔어!"라고 메시지를 던집니다.
    3.  **이벤트 수신 (Consumer):** 서버 내부의 컨슈머가 메시지를 낚아챕니다.
    4.  **푸시 알림 (SSE):** 컨슈머가 실시간 연결된 브라우저들에게 알림창을 띄우라고 신호를 보냅니다.
* **복습 포인트:** Kafka를 쓰는 이유는 '글 저장'과 '알림 전송'을 분리(Decoupling)하여 시스템의 부하를 줄이기 위함입니다.

### [Day 3] 실시간 채팅: WebSocket과 STOMP
새로고침 없이 대화를 주고받는 채팅 기능을 구축했습니다.

* **핵심 기술:** WebSocket, STOMP, SockJS
* **동작 원리:**
    * **WebSocket:** 한번 연결되면 끊어지지 않는 전용 파이프라인을 구축합니다.
    * **STOMP:** 이 파이프라인 안에서 "이건 1번 방 메시지야"라는 라벨을 붙여 데이터를 전송하는 규칙(프로토콜)입니다.
* **복습 포인트:** `subscribe`(구독)와 `publish`(발행)의 개념. 우체통에 편지를 넣으면(Publish), 그 우체통을 지켜보던 사람들(Subscribe)이 편지를 받는 구조입니다.

### [Day 4] 분산 아키텍처: 다중 서버 채팅 연동
서버가 10대, 100대로 늘어나도 채팅이 끊기지 않게 만드는 고난도 과정입니다.

* **핵심 문제:** 1번 서버 유저와 2번 서버 유저는 서버가 다르기 때문에 서로 대화가 안 됩니다.
* **해결책:** 모든 서버가 **중앙 우체국(Kafka)**을 바라보게 만듭니다.
* **동작 원리:** 유저 메시지 ➡️ 서버 1 ➡️ **Kafka** ➡️ 서버 1, 서버 2, 서버 3... ➡️ 각 서버에 접속한 유저들.
* **복습 포인트:** **동적 라우팅**. `/topic/room/{roomId}` 주소를 사용하여, Kafka에서 온 메시지를 해당 방 번호에 맞는 사람들에게만 정확히 골라 배달합니다.

### [Day 5] 데이터 영속성: 대화 기록 저장 및 로드
새로고침을 해도 예전 대화 내용이 그대로 남아있게 만드는 마무리 단계입니다.

* **핵심 기술:** MySQL (Chat 테이블), JPA Pagination (`findTop100`)
* **동작 원리:**
    1.  **저장:** 메시지가 서버에 들어오면 Kafka로 쏘기 전에 DB에 먼저 한 줄 저장합니다.
    2.  **로드:** 유저가 채팅방에 들어오는 순간, API가 작동하여 최근 100개의 대화를 DB에서 긁어와 화면에 먼저 그려줍니다.
* **복습 포인트:** `LocalDateTime.now()`를 통한 시간 관리. 대화의 순서가 꼬이지 않게 시간을 정확히 기록하는 것이 생명입니다.

## ⚠️ 3. 초보 개발자가 자주 하는 실수 (Troubleshooting)

1.  **오타 (Typo):** 주소의 슬래시 하나(`/`) 차이로 데이터가 전달되지 않는 경우가 많습니다. (예: `/topic/room1` vs `/topic/room/1`)
2.  **JPA 메서드 이름:** `findTopBy...`는 딱 1개만 가져옵니다. 여러 개를 가져오려면 `findTop100By...`처럼 숫자를 명시해야 합니다.
3.  **DB 데이터 타입:** 긴 텍스트를 저장할 때는 `@Column(columnDefinition = "TEXT")`를 사용해야 글자 수 제한에 걸리지 않습니다.
4.  **동시 연결 제한:** HTTP/1.1은 브라우저당 동시 연결 수가 약 6개로 제한됩니다. 탭을 너무 많이 열면 SSE나 WebSocket 연결이 멈출 수 있으니 테스트 시 주의하세요.

## 💡 4. 프로젝트 수정을 위한 "치트 시트"

* **새로운 알림 기능을 추가하고 싶다면?**
    * `dto` 패키지에 새로운 Event 클래스 생성
    * `KafkaConsumerService`에 새로운 `@KafkaListener` 메서드 추가
* **채팅방에 '인원수 제한' 기능을 넣고 싶다면?**
    * `ChatController`의 `addUser` 메서드에서 현재 해당 방의 `emitters`나 세션 수를 체크하는 로직 추가
* **검색 기능을 만들고 싶다면?**
    * `PostRepository`에 `findByTitleContaining` 메서드 정의 후 컨트롤러에서 호출
